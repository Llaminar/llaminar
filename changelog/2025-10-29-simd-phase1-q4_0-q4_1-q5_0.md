### Q5_0Tensor Implementation

**Files Modified**:
- `src/v2/tensors/Q5_0Tensor.cpp`
- `src/v2/tensors/Tensors.h`

**Key Changes**:

1. Same infrastructure pattern as Q4_0/Q4_1

2. Implemented `decodeBlockAVX512()`:
   - Q5_0 complexity: 4-bit low nibbles + 1-bit high bits
   - High bits stored in separate 32-bit `qh` field
   - Extracts high bits using bitwise operations and `_mm512_set_epi32()`
   - Combines: `(low_nibble | (high_bit << 4)) - 16`
   - Uses `_mm512_or_si512()` to merge high bits
   - Uses `_mm512_sub_epi32()` for -16 bias

3. Implemented `decodeBlockAVX2()`:
   - Processes in 4×8 chunks (vs Q4_0's 4×8, Q4_1's 4×8)
   - Extracts high bits per 8-value chunk from `qh` bytes
   - Uses `_mm256_set_epi32()` for per-element high bit extraction
   - More complex than Q4_1 due to separate high bit storage

**Performance Expectations**:
- Scalar: ~1.0× baseline
- AVX2: ~2-3× speedup (complex bit extraction overhead)
- AVX512: ~3-5× speedup (better amortization of bit extraction)

## Implementation Pattern

All Phase 1 tensors follow this standard pattern:

```cpp
// 1. Include headers
#include "../utils/DebugEnv.h"
#include "../utils/CPUFeatures.h"
#if defined(__AVX512F__)
#include <immintrin.h>
#endif
#if defined(__AVX2__)
#include <immintrin.h>
#endif

// 2. Runtime dispatch in decodeBlock()
void TensorType::decodeBlock(const Block &block, float *output) {
    const auto &env = debugEnv();
    
    if (env.dequant.simd_path == "scalar") {
        decodeBlockScalar(block, output);
        return;
    }
    
    #if defined(__AVX512F__)
    if (env.dequant.simd_path == "avx512" || 
        (env.dequant.simd_path == "auto" && cpu_supports_avx512())) {
        decodeBlockAVX512(block, output);
        return;
    }
    #endif
    
    #if defined(__AVX2__)
    if (env.dequant.simd_path == "avx2" || 
        (env.dequant.simd_path == "auto" && cpu_supports_avx2())) {
        decodeBlockAVX2(block, output);
        return;
    }
    #endif
    
    decodeBlockScalar(block, output);  // Final fallback
}

// 3. Scalar implementation (extracted from original)
void TensorType::decodeBlockScalar(const Block &block, float *output) {
    // Original scalar logic
}

// 4. AVX512 implementation
#if defined(__AVX512F__)
void TensorType::decodeBlockAVX512(const Block &block, float *output) {
    // Process 16 values at a time
}
#endif

// 5. AVX2 implementation
#if defined(__AVX2__)
void TensorType::decodeBlockAVX2(const Block &block, float *output) {
    // Process 8 values at a time
}
#endif
```

## Technical Details

### Q4_0 Decode Formula
- **Block**: 32 elements, FP16 scale, 16 bytes packed nibbles
- **Formula**: `output[i] = scale * (nibble - 8)`
- **SIMD Challenge**: Nibble extraction + broadcast scale + vectorized subtract

### Q4_1 Decode Formula
- **Block**: 32 elements, FP16 scale, FP16 min, 16 bytes packed nibbles
- **Formula**: `output[i] = scale * nibble + min`
- **SIMD Challenge**: Nibble extraction + FMA (scale*val+min)

### Q5_0 Decode Formula
- **Block**: 32 elements, FP16 scale, 4 bytes high bits, 16 bytes low nibbles
- **Formula**: `output[i] = scale * ((low_nibble | (high_bit << 4)) - 16)`
- **SIMD Challenge**: Combining separate high bits with low nibbles

### CPU Feature Detection

Uses existing `CPUFeatures.h` infrastructure:

```cpp
bool cpu_supports_avx512();  // Checks __AVX512F__ at compile + runtime CPUID
bool cpu_supports_avx2();    // Checks __AVX2__ at compile + runtime CPUID
```

Runtime detection ensures:
- No illegal instructions on older CPUs
- Graceful fallback to supported paths
- Testing on heterogeneous systems

## Lessons Learned

### Debug Environment Pattern Works Well

The `debugEnv()` singleton with lazy initialization provides:
- ✅ Single source of truth for environment variables
- ✅ No repeated `std::getenv()` calls in hot paths
- ✅ Easy to extend with new configuration knobs
- ✅ Cached values (no performance penalty)

### SIMD Path Testing Strategy

The 4-way dispatch (auto/scalar/avx2/avx512) enables:
- ✅ Forced path testing for numerical validation
- ✅ Performance regression detection (compare avx512 vs avx2)
- ✅ Fallback verification (ensure scalar always works)
- ✅ Bisection debugging (isolate which SIMD level has bugs)

### Extraction Pattern Is Clean

Extracting scalar code to separate method:
- ✅ Makes dispatcher logic clearer
- ✅ Preserves original implementation for reference
- ✅ Allows independent testing of scalar path
- ✅ Reduces code duplication in headers

### AVX512 Wins Are Real

16-wide operations vs 8-wide (AVX2):
- ✅ Fewer iterations (2 vs 4 for 32 elements)
- ✅ Better instruction throughput on modern CPUs
- ✅ Reduced overhead from setup/teardown
- ✅ Expected 2× speedup over AVX2 (on top of AVX2's 2-3× over scalar)
