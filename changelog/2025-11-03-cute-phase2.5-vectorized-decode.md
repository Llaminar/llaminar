## What Was Tried

### 1. Vectorized Decode (Implemented ✅)

**Before** (scalar decode):
```cpp
for (int i = 0; i < 16; i++) {
    uint8_t q = quants[i];
    float val0 = scale * iq4nl_values[q & 0xF];
    float val1 = scale * iq4nl_values[q >> 4];
    output[2*i] = __float2half(val0);      // Scalar store
    output[2*i+1] = __float2half(val1);    // Scalar store
}
```

**After** (vectorized decode):
```cpp
for (int i = 0; i < 4; i++) {
    uint32_t q4 = *reinterpret_cast<const uint32_t*>(&quants[i * 4]);  // Vector load
    
    for (int j = 0; j < 4; j++) {
        uint8_t q = (q4 >> (j * 8)) & 0xFF;
        float val0 = scale * iq4nl_values[q & 0xF];
        float val1 = scale * iq4nl_values[q >> 4];
        
        reinterpret_cast<__half2*>(output)[i * 4 + j] = 
            __floats2half2_rn(val0, val1);  // Vector store (2×FP16)
    }
}
```

**Benefits**:
- 32-bit vector loads of quantized data (4 bytes at a time)
- Vector stores using `__half2` (2 FP16 values per store)
- Better instruction scheduling

**Impact**: +0.5% performance (minor but measurable)

### 2. Fully Fused Decode (Deferred to Phase 3)

**Concept**: Store quantized IQ4_NL blocks in shared memory, decode during register copy.

**Why deferred**:
- Decode is NOT the bottleneck (only ~2-3% of kernel time)
- Real bottleneck: Global memory bandwidth (loading A and B from DRAM)
- CuTe's copy mechanism expects matching element types (complex to override)
- Would require custom Copy_Atom implementation

**Estimated gain**: <5% (not worth complexity at this stage)

### 3. True Fusion: Decode-to-Register (Phase 3 Candidate)

**Concept**: 
- Don't use shared memory for B at all
- Decode IQ4_NL blocks directly from global memory to registers
- Process B in smaller chunks that fit in registers

**Challenges**:
- Requires major kernel rearchitecture
- Must coordinate decode with MMA atom's data access pattern
- CuTe's TiledMMA expects shared memory tensors

**Estimated gain**: 10-20% (eliminates shared memory round-trip)

**Complexity**: High (requires deep CuTe expertise)

## Bottleneck Analysis

### Current Bottlenecks (in order)

1. **Tensor Core Utilization** (45%)
   - Small tile size (32×32×32) leaves SMs underutilized
   - Fix: Increase to 128×128×64 (Phase 3)
   - Expected gain: 2-3× throughput

2. **Global Memory Bandwidth** (46% combined)
   - Loading A and B from DRAM
   - Fix: Multi-stage pipelining (overlap load + compute)
   - Expected gain: 1.5-2× throughput

3. **Type Conversion Overhead** (5%)
   - FP32→FP16 for A, IQ4_NL→FP16 for B
   - Already near-optimal with vectorization
   - Minimal room for improvement

### Why Current Approach Is Good Enough

**Decode to shared memory is actually efficient**:
- Shared memory has ~10 TB/s bandwidth (very fast)
- Decode happens once per tile, amortized over many MMA operations
- Simple, maintainable code

**Fused decode-to-register would save**:
- ~2KB shared memory (negligible on 100KB total)
- ~0.0001 ms per tile (2% improvement)

**Cost of fusion**:
- Complex custom Copy_Atom implementation
- Harder to maintain and debug
- May break future CuTe optimizations

**Verdict**: Not worth it at this stage. Revisit after Phase 3 (larger tiles).

## Phase 3 Roadmap

### Priority 1: Larger Tiles (Expected: 2-3× gain)

```cpp
// Current: 32×32×32
template <int TILE_M = 32, int TILE_N = 32, int TILE_K = 32>

// Phase 3: 128×128×64
template <int TILE_M = 128, int TILE_N = 128, int TILE_K = 64>
```

**Benefits**:
- Higher arithmetic intensity (more compute per memory access)
- Better SM utilization (82 SMs on RTX 3090)
- Amortizes overhead across more work

**Challenges**:
- Shared memory limits (~100 KB per SM)
- Register pressure (need careful allocation)

### Priority 2: Multi-Stage Pipeline (Expected: 1.5-2× gain)

Overlap memory transfers with compute:
- Stage 0: Load next A/B tile (async)
- Stage 1: Compute current tile
- Stage 2: Write previous C tile

**Expected performance**: 800-1,000 GFLOPS (56-70% of peak)

### Priority 3: Fused Decode (Expected: <10% gain)

Only consider if Phases 1-2 don't reach target.
