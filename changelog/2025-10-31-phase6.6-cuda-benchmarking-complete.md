## Performance Analysis

### Why CPU is Slow (~2 GFLOPS constant)

The CPU implementation is a **simple nested loop GEMM** (reference implementation):
```cpp
void cpuGemm(const float* A, const float* B, float* C, int m, int n, int k) {
    for (int i = 0; i < m; ++i) {
        for (int j = 0; j < n; ++j) {
            float sum = 0.0f;
            for (int p = 0; p < k; ++p) {
                sum += A[i * k + p] * B[j * k + p];
            }
            C[i * n + j] = sum;
        }
    }
}
```

**Limitations**:
- No vectorization (AVX/SIMD)
- No tiling/blocking
- No multi-threading
- Cache-unfriendly access pattern

**Why this is OK for benchmarking**:
- Shows **raw CUDA kernel performance** vs baseline
- Highlights GPU's massive parallelism advantage
- Real V2 CPU implementation (with AVX512) would be 50-100× faster (~100-200 GFLOPS)
- Speedup would still be **2-5× in favor of CUDA** for large batches

### Why CUDA is Fast

**Kernel Optimizations**:
1. **Shared Memory Tiling**: 16×16 thread blocks with 16×32 shared memory tiles
2. **Coalesced Memory Access**: 16 threads load consecutive A elements (bandwidth-optimal)
3. **Block-Level Parallelism**: 82 SMs × 12 blocks/SM = 984 concurrent blocks
4. **Decode-Once Strategy**: Each IQ4_NL block decoded once, reused by 16 threads
5. **Register-Level Accumulation**: Dot product computed in registers (fast)

**Scalability Factors**:
- **Small matrices** (single token): Launch overhead dominates → 6-7× speedup
- **Medium matrices** (32-128 batch): Optimal occupancy → 130-210× speedup
- **Large matrices** (512-2048): Compute-bound → 220-230× speedup (peak)

### Throughput Ceiling Analysis

**RTX 3090 Theoretical Peak**: ~2000 GFLOPS (FP32)

**Our Performance**: 479 GFLOPS (24% of peak)

**Why not 100%?**
1. **IQ4_NL Decode Overhead**: Dequantization isn't free (~10-15% compute time)
2. **Memory Bandwidth**: Loading A and blocks from global memory (not purely compute-bound)
3. **Tile Size**: 16×16 blocks leave some ALUs idle on large SMs
4. **FP16 Conversion**: Scale factor conversion adds latency

**Potential Improvements** (Future Work):
- Larger tile sizes (32×32) for better occupancy → 20-30% speedup expected
- Fused decode into GEMM accumulation → 10-15% speedup
- Warp-level primitives (wmma) for tensor cores → 2-3× speedup possible
- BF16 accumulation → Reduce memory traffic, increase throughput

## Future Optimization Opportunities

### Short-Term (20-40% improvement expected)

1. **Larger Tile Sizes** (32×32 or 64×32)
   - Better SM occupancy
   - More work per thread block
   - Expected: +20-30% GFLOPS

2. **Prefetch to Shared Memory**
   - Hide global memory latency with double-buffering
   - Expected: +10-15% GFLOPS

3. **Warp-Level Reduction**
   - Use warp shuffle for final accumulation
   - Expected: +5-10% GFLOPS

### Medium-Term (2-3× improvement expected)

4. **Tensor Core Integration**
   - Use `wmma` (warp matrix multiply-accumulate)
   - Convert IQ4_NL → FP16 in registers
   - Expected: +100-200% GFLOPS (2-3× total)

5. **Fused Decode-GEMM**
   - Decode IQ4_NL blocks on-the-fly during accumulation
   - Eliminate separate decode step
   - Expected: +10-15% GFLOPS

### Long-Term (10× improvement possible)

6. **Ampere/Ada-Specific Optimizations**
   - Async copy from global to shared
   - Cluster-level execution
   - Expected: +30-50% GFLOPS

7. **Multi-GPU Sharding**
   - Partition large matrices across GPUs
   - MPI + NCCL for inter-GPU communication
   - Expected: Near-linear scaling (2 GPUs → 2× throughput)
