## Executive Summary

Created a Python generator system for CUDA GEMM kernel variants that generates `.inc` files containing LAUNCH_VARIANT macro calls. The system successfully generates 200 kernel configurations distributed across 10 include files, but there's a **parameter mapping bug** between the Python generator and the CUDA kernel template that needs to be fixed.

## What Was Accomplished

### 1. Python Generator Script (`generate_cuda_gemm_variants.py`)

**Created**: ~400-line Python script that:
- Generates 648 valid CUDA kernel configurations from 4050 total combinations
- Validates against hardware constraints (shared memory, registers, block size)
- Scores configurations using analytical performance model
- Selects top 200 configs for instantiation
- Distributes across 10 `.inc` files for organization

**Configuration Space**:
```python
TILE_M_VALUES = [16, 32, 64]
TILE_N_VALUES = [16, 32, 64]
TILE_K_VALUES = [32]  # Fixed for IQ4_NL

```

**Validation Logic**:
```python
# CRITICAL: Must match kernel static_assert
tile_m == block_threads_y * work_items_y  # ✅ Correct in code
tile_n == block_threads_x * work_items_x  # ✅ Correct in code
shared_mem <= 48KB (sm_70)
registers_per_thread <= 255
block_threads <= 1024
```

**Performance Scoring**:
- Arithmetic intensity (FLOP/byte ratio)
- Occupancy potential (threads per block)
- Work per thread (register efficiency)
- Prefetching effectiveness
- Vectorization benefits
- Tile aspect ratio balance

## Current Issue: Parameter Mapping Bug

### Problem

**Compilation Error**:
```
error: static assertion failed with "TILE_N must equal THREADS_N * WORK_N"
detected during instantiation of iq4nl_gemm_kernel_variant<..., TILE_N=32, THREADS_N=16, WORK_N=4, ...>
```

**Root Cause**: Incorrect parameter mapping in Python generator

The kernel template uses:
```cpp
template<
    int TILE_M, int TILE_N, int TILE_K,
    int THREADS_M, int THREADS_N,  // M=rows (Y), N=cols (X)
    int WORK_M, int WORK_N,
    ...
>
```

The Python generator currently uses:
```python
BLOCK_THREADS_X_VALUES = [8, 16]  # Should map to THREADS_N
BLOCK_THREADS_Y_VALUES = [8, 16]  # Should map to THREADS_M
WORK_ITEMS_X_VALUES = [1, 2, 4, 8]  # Should map to WORK_N
WORK_ITEMS_Y_VALUES = [1, 2, 4, 8]  # Should map to WORK_M
```

**The bug**: When generating configs, the loop does:
```python
for block_threads_x in BLOCK_THREADS_X_VALUES:  # N dimension
    for block_threads_y in BLOCK_THREADS_Y_VALUES:  # M dimension
        for work_items_x in WORK_ITEMS_X_VALUES:  # N dimension
            for work_items_y in WORK_ITEMS_Y_VALUES:  # M dimension
                config = (tile_m, tile_n, tile_k,
                         block_threads_x, block_threads_y,  # ❌ WRONG ORDER!
                         work_items_x, work_items_y,        # ❌ WRONG ORDER!
                         ...)
```

Should be:
```python
config = (tile_m, tile_n, tile_k,
         block_threads_y, block_threads_x,  # ✅ M first, then N
         work_items_y, work_items_x,        # ✅ M first, then N
         ...)
```

## Design Decisions

### Why .inc Files Instead of .cu Files?

**Considered**: Generating separate .cu files (like CPU pattern)

**Chosen**: .inc files included in main CudaGemmVariants.cu

**Rationale**:
1. ✅ **Simpler**: No need for explicit template instantiation syntax
2. ✅ **Faster builds**: Single nvcc compilation vs 50+ parallel
3. ✅ **Macro context**: LAUNCH_VARIANT macro available in scope
4. ✅ **Less boilerplate**: No namespace, includes, force-link stubs
5. ❌ **Trade-off**: Larger single compilation unit (slower incremental builds)

**Verdict**: .inc approach is cleaner for CUDA template instantiation

### Why Top 200 Configs?

**Total valid**: 648 after strict validation

**Selected**: 200 via analytical performance model

**Rationale**:
1. ✅ **Manageable compilation time**: 200 instantiations compile in ~30s
2. ✅ **Good coverage**: Spans small/medium/large tiles, prefetch on/off, transpose on/off
3. ✅ **Auto-tuner bandwidth**: Benchmarking 5-10 per shape is reasonable
4. ❌ **May miss edge cases**: Some niche configs excluded

**Future**: Could generate all 648, but diminishing returns

## Lessons Learned

1. **CUDA template instantiation differs from CPU**:
   - Can't use explicit instantiation syntax for `__global__` functions
   - Must instantiate via kernel launch (implicit)
   - .inc files with macro calls work well

2. **Parameter naming matters**:
   - X/Y vs M/N confusion caused subtle bug
   - Kernel uses THREADS_M/THREADS_N (M=rows, N=cols)
   - Python should match kernel naming exactly

3. **Validation must match kernel assertions**:
   - `is_valid_config()` must check `tile_m == threads_m * work_m`
   - Not just `tile_m % (threads_m * work_m) == 0`
   - Static assertions caught this early

4. **CMake integration is straightforward**:
   - `execute_process()` runs generator at configure time
   - Dependency tracking via script timestamp
   - .inc files don't need CMake changes (just #include)
