## Root Cause Analysis

### The Problem

The original test helpers used naive bit manipulation for FP16 conversion:

```cpp
// WRONG: Just shifts bits (doesn't preserve IEEE 754 format)
d_fp16 = (u.i >> 16) & 0xFFFF;
```

This approach:
1. ❌ Doesn't preserve sign bit correctly
2. ❌ Doesn't adjust exponent bias (FP32 uses 127, FP16 uses 15)
3. ❌ Doesn't round mantissa bits properly
4. ❌ Produces garbage FP16 values

**Example**:
```
FP32: 0.00123456  (0x3AA16000 in hex)
Naive shift: 0x3AA1 → Interprets as ~0.63 in FP16 (WRONG!)
Proper conversion: 0x2508 → 0.00123 in FP16 (CORRECT)
```

### The Solution

V2's `FP16Utils.h` provides proper IEEE 754 conversion:

```cpp
uint16_t fp32_to_fp16(float f);  // Correct exponent bias, mantissa rounding
float fp16_to_fp32(uint16_t h);  // Handles denormals, infinities, NaNs
```

**Features**:
- ✅ Proper exponent bias adjustment (127 ↔ 15)
- ✅ Round-to-nearest-even mantissa rounding
- ✅ Handles special values (denormals, infinities, NaNs)
- ✅ Clamps overflow to ±infinity

## Lessons Learned

1. **Never use naive bit manipulation for floating-point conversion**
   - IEEE 754 formats have complex encoding (sign, exponent bias, mantissa)
   - Use library functions or hardware intrinsics

2. **Reuse existing utilities**
   - V2 already had proper FP16 conversion in `FP16Utils.h`
   - Avoid reinventing the wheel (especially for complex formats)

3. **Simple tests reveal issues faster**
   - `debug_cuda_gemm.cu` with known values isolated the problem quickly
   - Complex random tests hid the root cause initially

4. **FP16 requires special care**
   - Different exponent bias (127 vs 15)
   - Mantissa precision loss (23 bits → 10 bits)
   - Special value handling (denormals, infinities, NaNs)
